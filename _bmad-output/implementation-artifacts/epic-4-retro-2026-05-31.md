## Cross-Epic Pattern Analysis

### Recurring Strengths

1. **100% green builds**: 0 TypeScript errors at close across all 3 stories. The build discipline is unbroken across all 4 epics (12 stories total).

2. **Service layer: 100% compliant**: No `fetch` calls in components or hooks across the entire epic. NFR-6 has never been violated.

3. **Blueprint-following = fewer patches**: The epic averaged 1.7 patches/story vs 4 patches/story in Epic 3. Better Dev Notes = less review work.

4. **Model agnosticism confirmed**: Story 4.3 (Sonnet 4.6) was cleaner than 4.1/4.2 (Opus 4.8). Model quality is a function of blueprint quality, not model choice.

5. **Component reuse working**: `LevelMetricCard`, `RecommendationDisplay`, `DomainTag`, `MLFieldDisplay` from Epic 3 were cleanly reused in 4.3's `MetricDetailTable` with no modification.

### Recurring Challenges

1. **Same bug, consecutive stories (4.1→4.2)**: 'Missing fallback on Record lookup' appeared as a patch in 4.1 (`globalTrend`) and again in 4.2 (`trend`). The 4.1 review categorized it as a one-off fix, not a pattern. Cross-story pattern propagation is not happening.

2. **23 deferred items with identical signatures**: At least 8/23 items are the UTC timezone off-by-one. This is an infrastructure-level debt being re-documented as a story-level defer, which inflates the count and obscures true risk.

3. **`navigate(-1)` design error in blueprint**: The review catch in 4.3 was not an implementation mistake — the Dev Notes specified `navigate(-1)`. The architect blueprint contained an incorrect assumption about navigation entry points. Blueprint review needs an "entry point safety" checklist.

4. **`badge-red` CSS class absent**: The 4.1 review patched around a missing CSS class with an inline style. This means the design system in `src/index.css` is incomplete relative to the original `NeurologistDashboard.tsx` reference design.

## What Was Challenging

- Record lookup fallback missed twice in consecutive stories (4.1 → 4.2)
- `navigate(-1)` design error in blueprint passed through review unchallenged

## Key Takeaways

1. **Organizational forcing functions, not good intentions**: B1/B2 need to be the *first task in the first story they enter*, not a footnote in a retrospective. If a critical fix has no named owner and no explicit story placement, it will not happen.

2. **Pattern propagation from review must be explicit**: When a review catches 'missing fallback on Record lookup', the fix must generate a carry-forward rule in the next story's Dev Notes — not just patch the current file.

3. **Blueprint entry-point safety**: Every page component (`SessionDetailPage`, any future standalone page) must document whether it can be a navigation entry point. If yes, `navigate(-1)` is wrong; a named path is required.

4. **Design system completeness is a feature requirement**: The visual gap between `NeurologistDashboard.tsx` (reference) and the current implementation is significant. The old design had tooltips, custom chart dots, session row layout, sidebar sections, and notification patterns that are absent or incomplete in the new system.

5. **Model agnosticism is confirmed**: Blueprint quality determines implementation quality, not model choice.
